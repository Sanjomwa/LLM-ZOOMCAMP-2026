# Module 4 (Evaluation) — FAQ Additions

These are questions that came up while working through the Module 4 homework — specifically around ground-truth generation, the evaluation harness, and comparing text/vector/hybrid search on real numbers. None of these touch graded answers, just the concepts and reasoning behind each step. Posting here in case others hit the same questions.

## Q: Why does the homework only generate questions for 3 pages in Q1 instead of all 72?
A: Generating ground truth with an LLM costs real money per call, and the homework wants you to know that cost before committing to the full batch, not after.

Checking token usage on 3 pages first gives you a per-page cost estimate you can multiply out: if 3 pages average ~1,350 input tokens each, 72 pages is a predictable, budgetable number rather than a surprise on the invoice. This is the same discipline the module's batch-generation lesson demonstrates — track cost from the first call, not after the full run finishes.

## Q: Why use the provided ground-truth.csv instead of generating all 72 pages yourself?
A: Two reasons, both about reproducibility. First, cost — generating 360 questions across 72 pages with retries is meaningfully more expensive than the 3-page sample. Second, and more important: LLM generation isn't fully deterministic, so regenerating your own ground truth — even with the identical prompt and process — produces a different dataset than the one the expected multiple-choice answers were computed against. Using the frozen, provided CSV is what makes your Hit Rate/MRR numbers comparable to everyone else's.

## Q: Why does the ground-truth generation prompt explicitly say not to reuse the lesson page's own wording?
A: Because an LLM generating a question from a document naturally echoes some of that document's vocabulary, even when told not to — and that inflates retrieval metrics relative to how real users actually ask questions. Real users paraphrase; they don't usually re-type your documentation's exact phrasing back at you.

This instruction doesn't fully eliminate the bias (see the next question), but skipping it entirely would make the ground truth even easier than real queries, and the resulting Hit Rate/MRR numbers would overstate how good retrieval actually is in production.

## Q: Why did keyword (text) search outperform vector search on this dataset?
A: This was the most counter-intuitive result of the whole homework — most people (myself included, going in) expect vector search to win by default. It didn't, on either Hit Rate or MRR.

The likely explanation ties back to the previous question: these ground-truth questions were generated by an LLM reading the lesson page directly, and despite explicit instructions to paraphrase, some source vocabulary still leaks through. That keeps keyword overlap between question and source chunk unusually informative for this specific dataset — an advantage keyword search doesn't reliably get with real, independently-phrased user questions. It's a solid reminder that "vector search is more sophisticated" is not the same claim as "vector search will score higher on your data," and the only way to know which is true is to actually run the evaluation.

## Q: Why does evaluate() only need a one-line change to work with this homework's ground truth?
A: The module's own compute_relevance, hit_rate, mrr, and evaluate functions were written generically against a search_function parameter, with the relevance check as the only place that touches the ground truth's specific shape. The module's FAQ ground truth matched on id; this homework's lesson-page ground truth matches on filename. Swap that one comparison, and the exact same four functions correctly score text_search, vector_search, and hybrid_search — three structurally different retrieval methods — with nothing else touched. That's the payoff of writing evaluation code against an interface instead of a specific index from the start.

## Q: Why sweep RRF's k instead of just using the paper's default of 60?
A: Because the "right" k is a property of the specific corpus and query distribution, not a universal constant — the same reasoning the module applies to field boosts. On this dataset, k=1 produced the best MRR by a clear margin, while k=50, k=100, and k=200 were identical to four decimal places. That plateau is itself informative: past a certain point, k stops changing which chunk lands on top for this data, so k=60 would have landed in an unremarkable flat region here rather than at an optimum. You only find that out by measuring across the range, not by trusting the default.

## Q: Why does the notebook download the embedding model itself instead of assuming it's already there?
A: Learned this one the hard way. The homework instructions say to "keep working in the same project" as Homework 2 — which is correct advice, but it means the notebook silently depends on embedder.py and a ~90MB downloaded ONNX model already sitting in that project's directory. Move or copy the notebook anywhere else (a different folder, a fresh clone, a different machine) and from embedder import Embedder throws ModuleNotFoundError, and even after fixing that import, Embedder() itself fails with a file-not-found error if the model was never downloaded into the new location.

The fix that actually holds up: have the notebook fetch embedder.py, download.py, and run the download itself, using wget -nc (no-clobber) so it's a no-op if the files already exist. That makes the notebook portable and reproducible on its own, rather than silently coupled to wherever Homework 2 happened to be run.

## Q: Why does the same evaluate() call take noticeably longer for hybrid search than for text or vector search alone?
A: hybrid_search isn't a third independent index — it calls both text_search and vector_search internally (10 results each) and merges them with RRF. So every one of the 360 ground-truth questions triggers two searches instead of one whenever evaluate(ground_truth, hybrid_search) runs, and the k-sweep in Q6 repeats that for four different k values. The cost is entirely from re-running the same two underlying searches, not from RRF itself, which is a cheap ranking merge once both result lists exist.
